# Vanilla 4.4 ‚Äî Economy Architecture

> Verified against Stellaris 4.4.3 (Pegasus). Nomads DLC notes are marked **(Nomads DLC)**.

## Orbital Deposits (Space Resources)

### File Location
`common/deposits/01_orbital_deposits.txt` ‚Äî primary orbital deposits (minerals, energy, research, strategic resources)

Also: `02_sr_deposits.txt` (strategic resources), `03_leviathans_deposits.txt`, `04_distant_stars_deposits.txt`, and many DLC-specific deposit files (now `05`‚Äì`17`, including `17_nomads_deposits.txt` ‚Äî see Nomads note below).

### Structure

Each orbital deposit has:
- `resources` ‚Äî category and production (`produces = { minerals = X }`)
- `station` ‚Äî station class required (e.g., `shipclass_mining_station`)
- `is_for_colonizable` ‚Äî whether it appears on habitable worlds
- `potential` ‚Äî spawn conditions
- `habitat_modifier` ‚Äî bonus districts when habitat is built over it
- `drop_weight` ‚Äî spawn probability with conditional modifiers

### Economic Category

Orbital deposits use `orbital_mining_deposits` and `orbital_research_deposits` categories.

**Category hierarchy** (from `common/economic_categories/00_common_categories.txt`):
```
stations
  ‚îî‚îÄ station_gatherers          (line 684 ‚Äî generates produces_mult, cost_mult)
  ‚îÇ    ‚îî‚îÄ orbital_mining_deposits   (line 1026, parent = station_gatherers)
  ‚îÇ         ‚îî‚îÄ hab_mining_deposits  (line 1034, parent = orbital_mining_deposits)
  ‚îî‚îÄ station_researchers        (line 694 ‚Äî generates produces_mult)
       ‚îú‚îÄ orbital_research_deposits (line 1030, parent = station_researchers)
       ‚îî‚îÄ hab_research_deposits     (line 1038, parent = station_researchers ‚Äî NOT orbital_research_deposits)
```
Line numbers per 4.4.3 (the whole file shifted down vs 4.3). Note `hab_research_deposits` hangs directly off `station_researchers`, not off `orbital_research_deposits`.

### Key Modifier for Scaling Space Resources

`station_gatherers_produces_mult` ‚Äî multiplier on ALL mining station output. Generated by the `station_gatherers` economic category. Can be applied via technologies, traditions, edicts, static modifiers, etc.

Similarly: `station_researchers_produces_mult` for research stations.

These are the **cleanest levers** for globally scaling space resource production.

> **‚öÝÔ∏è Mining stations gather BOTH minerals AND energy (verified 2026-06-27).** Energy orbital
> deposits (`d_energy_*`) use `category = orbital_mining_deposits` + `station = shipclass_mining_station`
> ‚Äî **identical to minerals**. So `station_gatherers_produces_mult` lifts mineral AND energy gathering
> together; there is **no separate energy-station modifier or tech**, and none is needed. (Strategic
> orbital deposits also ride `orbital_mining_deposits`, so they inherit this too ‚Äî see Track 3.)
> Research is the only separate one: `d_physics/society/engineering_*` ‚Üí `orbital_research_deposits` /
> `shipclass_research_station`, scaled by `station_researchers_produces_mult`.

> **Finite vanilla station-tech ramp:** `tech_space_mining_1..5` (engineering) each grant
> `station_gatherers_produces_mult +0.10` (+50% by tier-3 completion); `tech_space_science_1..5`
> (physics) likewise for `station_researchers_produces_mult`. Chains: 1‚Äì2 = tier 1, 3‚Äì4 = tier 2,
> 5 = tier 3. After the tier-3 end there is NO further finite station scaling until repeatables ‚Äî
> the "dead zone" `economy_overhaul` lever #6 fills by making these per-tier values escalate.

### Vanilla Orbital Yield Tiers (4.4.3)

Deposit IDs are tiered `d_<resource>_<N>` where higher N = richer:
- Minerals: `d_minerals_1` ‚Ä¶ `d_minerals_10`
- Energy: `d_energy_1` ‚Ä¶ `d_energy_10`
- Research: `d_physics_1`‚Ä¶`d_physics_10`, `d_society_1`‚Ä¶`d_society_15`, `d_engineering_1`‚Ä¶`d_engineering_10` (note: research deposits are `d_physics_N` / `d_society_N` / `d_engineering_N`, NOT `d_*_research_N`)
- Strategic resources (in `02_sr_deposits.txt`): tiered `d_exotic_gases_1‚Ä¶5`, `d_rare_crystals_1‚Ä¶5`, `d_volatile_motes_1‚Ä¶5`, etc.

(Tier counts verified 4.4.3; per-deposit yield numbers omitted per doc policy. The high tiers already exist ‚Äî see Scaling note.)

### Scaling Approach

To make space resources outproduce planets:
1. **Buff deposit yields** ‚Äî edit `01_orbital_deposits.txt` to increase base production
2. **Tech scaling** ‚Äî create technologies that apply `station_gatherers_produces_mult`
3. **Era-based scaling** ‚Äî use `on_yearly_pulse` events with game-year checks to upgrade deposits or apply modifiers
4. **New rich deposits** ‚Äî vanilla already ships tiers up to `d_minerals_10` / `d_society_15`; reuse/buff those before adding new IDs, or add `d_minerals_11+` with higher yields and appropriate spawn weights

### Nomads DLC ‚Äî space/waystation economy (**Nomads DLC**)

The Nomads DLC (4.4.x) adds a mobile, space-based economy layer that intersects this doc's scope:
- `common/deposits/17_nomads_deposits.txt` ‚Äî nomad/cruise deposits (e.g. `d_protostar`, `d_compact_modules`, `d_dark_matter_monolith`, deep-sleep cryo-pod deposits).
- `common/pop_jobs/17_nomads_jobs.txt` ‚Äî nomad jobs (e.g. `cruise_passenger`), gated on `has_origin = origin_forever_cruise` / cruise living standards.

DLC-gated ‚Äî only present/active with the Nomads DLC. Treat any mod hooks here as optional and guard them so non-DLC players are unaffected.

## Districts

### File Location
`common/districts/00_urban_districts.txt` and related files in `common/districts/`:
`00_special_districts.txt`, `01_arcology_districts.txt`, `02_rural_districts.txt`, `03_habitat_districts.txt`, `04_ringworld_districts.txt`, `05_wilderness_districts.txt`, `06_swap_districts.txt`, `07_ark_districts.txt`. See `00_DOCUMENTATION.txt` in that folder.

### Structure

Each district has:
- `base_buildable` ‚Äî whether it appears by default
- `min_for_deposits_on_planet` / `max_for_deposits_on_planet` ‚Äî spawn range
- `resources` ‚Äî upkeep and production
- `planet_modifier` ‚Äî modifiers when built (housing, amenity, etc.)
- `triggered_planet_modifier` ‚Äî conditional modifiers
- Job assignments (e.g., `job_miner_add = 2`)

### Key Levers for Nerfing Planetary Output

1. **Reduce jobs per district** ‚Äî change `job_X_add` values
2. **Increase district upkeep** ‚Äî modify `resources.upkeep` block
3. **Cap district slots** ‚Äî `planet_max_districts_add` / `planet_max_districts_mult` modifiers, also `DEFAULT_MAX_DISTRICTS_PER_PLANET = 4` in defines (line 1632 in 4.4.3)
4. **Reduce job output** ‚Äî modify individual jobs in `common/pop_jobs/`

## Planet Size

### Defines (`common/defines/00_defines.txt`)
- `DEFAULT_MAX_DISTRICTS_PER_PLANET = 4` (line 1632 in 4.4.3)

### How Planet Size Affects Districts
Planet size determines base district slots. Modifiable via:
- `planet_max_districts_add` ‚Äî modifier to add/remove district slots
- Planet class definitions in `common/planet_classes/` control size ranges and spawn weights

### Capping Planet Size
`@habitable_planet_max_size` (`00_scripted_variables.txt`, =25) appears in the planet-class
`planet_size = { min ‚Ä¶ max ‚Ä¶ }` ranges, BUT **changing it does NOT cap procedurally-generated
galaxy worlds** ‚Äî verified in-game 2026-06-24: with the var at 18, ordinary desert/tropical/tomb
worlds still spawned >18. (The home planet uses a *separate* `@homeworld_max_size`; gas giants and
special systems use explicit sizes.) **The working method is an `on_game_start` resize event** that
loops planets and shrinks oversized ones ‚Äî see `economy_overhaul/events/econ_overhaul_events.txt`
(`econ_overhaul.2`): `every_system = { every_system_planet = { limit = { is_colonizable = yes
is_capital = no planet_size > 18 } ‚Ä¶ set_planet_size ‚Ä¶ } }`. Resizing at game start catches all
natural worlds; capitals are excluded (shrinking a pop/district-laden homeworld causes overcrowding).

## ‚öÝÔ∏è Override Mechanics ‚Äî Verified Gotchas (in-game, 4.4.3, 2026-06-23/24)

Hard-won during the `economy_overhaul` in-game tests. "The file loads" ‚âÝ "the override wins."

1. **Scripted variables (`@var`) CANNOT be overridden by redefinition.** They are first-definition-
   wins in a flat global namespace; a mod always loads after vanilla, so a redefinition is rejected
   (`error.log`: `Variable name X is already taken`) and **vanilla keeps its value**. A `zzz_`-prefixed
   redefinition file does nothing. The ONLY working method is a **whole-file replacement** (mod ships
   a file with the same name as the vanilla one ‚Üí vanilla file isn't loaded ‚Üí no duplicate). Verbatim-
   copy the vanilla file, change only the target value. Cost: high conflict surface + must re-sync on
   version bumps.
2. **on_actions reject a bare `effect = {}` block** (`error.log`: `Unexpected token: effect`; vanilla
   uses `effect` in on_actions 0 times). To run an effect at an on_action, use `events = { <id> }` +
   a `hide_window = yes`, `is_triggered_only = yes` event whose `immediate = {}` does the work.
3. **DB objects (civics, techs, buildings, etc.) and defines DO override by last-loaded-wins** ‚Äî a
   same-key definition in a later-loading mod replaces vanilla's (`error.log` notes `‚Ä¶already exists,
   using the one at <your file>`). This asymmetry is why civic/tech overrides worked while scripted-
   variable redefinitions silently failed in the same mod.
4. **Local dev-mod deployment:** an external **absolute** `path` in the descriptor registers the mod
   (`ready_to_play`) but the engine never loads its content (`setup.log` has no trace). Use a
   directory **junction** into the game `mod/` folder + a **relative** descriptor `path="mod/<name>"`.
5. **`every_system_planet` is SYSTEM-scoped, not global.** Calling it from a scopeless event root
   iterates NOTHING (no error, no effect ‚Äî a silent no-op). To touch every planet in the galaxy:
   `every_system = { every_system_planet = { ‚Ä¶ } }` (vanilla `game_start.2` pattern).

## ‚öÝÔ∏è Pop generation & housing (4.4 ‚Äî verified 2026-06-24)

- **Pop pressure is HOUSELESSNESS, not joblessness.** Jobless pops become low-stakes **civilians**
  (no growth penalty); **overcrowding** stops growth at `OVERCROWDING_NO_GROWTH_THRESHOLD = 1.15` and
  declines pops at `1.25` (`NPop` defines; machine variants stop *assembly* at the same points). So to
  make "wide > tall" bite, **cut housing toward scarcity** ‚Äî a surplus does nothing.
- **City/urban districts grant ZERO jobs** (`district_city/hive/nexus` = housing only, 1000/1200/1100).
  **Specialist jobs come from ZONES** (`@scaling_district_*_job` in `100_scripted_variables_zones.txt`),
  not the district. To cut specialist *volume*, cut those zone vars; to cut city *housing*, edit the
  district `planet_housing_add` literals (whole-file replace `00_urban_districts.txt` ‚Äî no variable).
- **Assembly vs growth are different channels.** Mechanical/robot/machine pops use **assembly**
  (`planet_pop_assembly_add`, scaled by `planet_pop_assembly_mult`). But **hive spawning pools / clone
  vats / offspring nests produce *Monthly Organic Pop GROWTH*** (the spawning-drone job converts food ‚Üí
  growth), NOT assembly ‚Äî so `planet_pop_assembly_mult` does **not** touch them. `BASE_POP_ASSEMBLY = 0`
  (all assembly is job-sourced), and the `planet_pop_assemblers` economic category has its `produces`
  mult **commented out** in vanilla, so there is **no** `planet_pop_assemblers_produces_mult`. To nerf
  organic spawning growth, override the drone jobs' growth output (a broad `bonus_pop_growth_mult` also
  hits trait bonuses like Fertile).

## Economic Categories System

### File Location
`common/economic_categories/00_common_categories.txt`

### How It Works

Economic categories form a hierarchy. Each category can `generate_mult_modifiers` and `generate_add_modifiers` for `produces`, `cost`, and `upkeep`. This auto-generates modifiers like `category_name_produces_mult` that can be used anywhere modifiers are accepted.

### Key Categories for Our Mods

Line numbers are 4.4.3.

| Category | Line | Parent | Generated Modifier |
|----------|------|--------|-------------------|
| `station_gatherers` | 684 | `stations` | `station_gatherers_produces_mult`, `_cost_mult` |
| `station_researchers` | 694 | `stations` | `station_researchers_produces_mult` |
| `orbital_mining_deposits` | 1026 | `station_gatherers` | inherits parent modifiers |
| `orbital_research_deposits` | 1030 | `station_researchers` | inherits parent modifiers |
| `planet_structures` | 854 | `planets` | `planet_structures_cost_mult`, `_upkeep_mult` |
| `planet_buildings` | 864 | `planet_structures` | `planet_buildings_produces_mult`, `_cost_mult`, `_upkeep_mult` |
| `planet_deposits` | 1014 | `planets` | `planet_deposits_produces_mult` |

Note: there is **no** `planet_jobs` economic category in 4.4.3 ‚Äî pop-job production scales via the job-tier categories in `common/economic_categories/01_job_categories.txt` and via `planet_buildings`/`planet_structures`, not a `planet_jobs_produces_mult`.

Child categories inherit parent modifiers. Buffing `station_gatherers_produces_mult` affects all mining stations including habitats.

### Approach: More Concentrated, Less Frequent
- Reduce `drop_weight` for strategic deposits across most systems
- Create new "rich" strategic deposits with high yields (e.g., 5-8 gases)
- Adjust `drop_weight` modifiers to concentrate them in fewer systems
- This makes specific systems strategically critical ‚Äî worth fighting wars over

## Housing & Amenities

### Housing Deficit
When housing is insufficient, pops receive happiness and growth penalties. Severity scales with deficit size.

### Amenities Deficit
Amenity shortfalls reduce stability and happiness. Can be tuned via:
- `planet_amenities_no_happiness_add` ‚Äî amenities deficit threshold modifiers
- Static modifiers for deficit penalties in `common/static_modifiers/`

### Design Goal: Increase Deficit Penalties
Modify the static modifiers that define what happens at various deficit levels to make them harsher. This punishes overloaded mega-planets and rewards spreading pops across multiple worlds.
